# Terraform 環境分離（dev/prod）戦略ガイド

開発環境（dev）と本番環境（prod）を安全かつ効率的に管理・運用するためのTerraformのディレクトリ構成および実行手順を定義します。

## 1. なぜ環境分離が必要か？

Terraformは `terraform.tfstate` というファイルに「現在構築されているインフラの状態」を記憶しています。
変数（`.tfvars`）を分けるだけでは、この `tfstate` が同じ場所（同じS3パスなど）に保存されてしまい、「dev環境を作った後にprod環境を作ろうとすると、dev環境が削除・上書きされる」という重大な事故に繋がります。

そのため、**「変数」だけでなく「状態ファイル（tfstate）」も環境ごとに完全に分離する**必要があります。

## 2. 環境分離のアプローチ（ディレクトリ分離型）

本プロジェクトでは、実務で最も事故が起きにくく、管理がしやすい**「ディレクトリ分離型」**を採用（推奨）します。

共通化できるインフラ構成は `modules/` に切り出し、環境ごとのディレクトリからそれを呼び出すことで、DRY（コードの重複排除）と安全性を両立させます。

[Image of Terraform directory structure for environment separation]

### ディレクトリ構成案

```text
project-root/
├── environments/            # 環境ごとの実行ディレクトリ
│   ├── dev/
│   │   ├── backend.tf       # dev用のtfstate保存先（S3パス等）を指定
│   │   ├── main.tf          # modulesを呼び出し、dev用の変数を渡す
│   │   └── terraform.tfvars # dev環境特有の変数値
│   └── prod/
│       ├── backend.tf       # prod用のtfstate保存先（S3パス等）を指定
│       ├── main.tf          # modulesを呼び出し、prod用の変数を渡す
│       └── terraform.tfvars # prod環境特有の変数値
└── modules/                 # 共通化されたインフラの設計図
    ├── dynamodb/
    │   ├── main.tf
    │   └── variables.tf
    └── api_gateway/
        ├── main.tf
        └── variables.tf
```

## 3. 実行手順（ディレクトリ分離型の場合）

環境ごとにディレクトリが独立しているため、作業したい環境のディレクトリに移動してからコマンドを実行します。

### 開発環境（dev）にデプロイする場合

```bash
# 1. dev環境のディレクトリに移動
cd environments/dev

# 2. 初期化（dev専用のtfstateの向き先が設定される）
terraform init

# 3. 変更内容の確認（自動的に同ディレクトリの terraform.tfvars が読み込まれる）
terraform plan

# 4. デプロイの実行
terraform apply
```

補足: Workspace機能について
Terraformには1つのディレクトリのまま裏側で tfstate を切り替える terraform workspace という機能もあります。
手軽ですが、「今自分がどの環境（ワークスペース）にいるか」をターミナル上で常に意識しないと、誤って本番環境を操作するリスク（ヒューマンエラー）が高いため、本番運用を見据えた構成では「ディレクトリ分離型」を推奨としています。
