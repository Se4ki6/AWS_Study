S3 単体での公開と、CloudFront を組み合わせた公開には大きな違いがあります。一言で言うと、**「とりあえず見られれば良い（個人・検証用）」**のが S3 単体、**「世界中に高速・安全に届けたい（本番・商用）」**のが CloudFront 構成です。

主な違いを比較表とポイントにまとめました。

---

### S3 単体 vs CloudFront + S3 比較

| 比較項目         | S3 単体 (静的ウェブサイトホスティング) | CloudFront + S3 (CDN 構成)                       |
| ---------------- | -------------------------------------- | ------------------------------------------------ |
| **表示速度**     | バケットがあるリージョンから遠いと遅い | 世界中のエッジロケーションから配信され**高速**   |
| **セキュリティ** | バケットを全世界に公開する必要がある   | **OAC**で S3 を隠し、CF 経由のみに制限可能       |
| **HTTPS (SSL)**  | 非対応（独自ドメインの場合）           | **対応**（ACM を使って無料で HTTPS 化可能）      |
| **コスト**       | 比較的安価だが、アクセス増で転送量増   | 転送量単価が安く、キャッシュにより S3 費用を抑制 |
| **SEO**          | 速度や HTTPS の面で不利                | 高速かつ HTTPS 対応のため**有利**                |

---

### CloudFront を通す 3 つの大きなメリット

#### 1. 圧倒的なパフォーマンス向上（キャッシュ）

S3 単体の場合、アメリカのユーザーが東京リージョンの S3 にアクセスすると物理的な距離の分だけ遅延が発生します。
CloudFront を使うと、世界中にある**エッジロケーション**にデータが一時保存（キャッシュ）されるため、ユーザーは最寄りの拠点からデータを取得でき、表示が劇的に速くなります。

#### 2. セキュリティの強化 (OAC の利用)

S3 単体公開では、バケットの設定を「公開（パブリックアクセス）」にする必要があり、URL を知っていれば誰でも直接 S3 にアクセスできてしまいます。
CloudFront と**OAC (Origin Access Control)** を組み合わせれば、**「S3 自体は非公開にし、CloudFront からのリクエストしか受け付けない」**という強固な構成が可能です。

#### 3. 独自ドメインでの HTTPS 化

現在、Web サイトの HTTPS 化（常時暗号化）は必須です。
S3 単体では独自ドメインに対して HTTPS を適用できませんが、CloudFront を使えば AWS Certificate Manager (ACM) で発行した証明書を簡単に紐付けることができます。

> **💡 注意ポイント**
> CloudFront を導入すると「キャッシュ」が効くようになります。そのため、HTML ファイルを更新しても、古い内容がしばらく表示され続けることがあります。その場合は「キャッシュの無効化（Invalidation）」という操作が必要になることを覚えておくと役立ちます。
