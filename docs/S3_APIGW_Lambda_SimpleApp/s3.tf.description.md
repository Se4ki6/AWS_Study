# s3.tf - S3バケット設定の解説

## 概要
このファイルは、WebサイトのフロントエンドをホスティングするためのS3バケットとその関連リソースを定義しています。静的ウェブサイトホスティング機能を使用して、HTML/CSS/JavaScriptなどの静的コンテンツを公開します。

## リソース構成

### 1. random_id.bucket_suffix
```terraform
resource "random_id" "bucket_suffix" {
  byte_length = 4
}
```
**目的**: S3バケット名の一意性を確保するためのランダムなサフィックスを生成

- `byte_length = 4`: 4バイト（8文字の16進数）のランダムな文字列を生成
- S3バケット名はグローバルで一意である必要があるため、衝突を避けるために使用

**例**: `my-app-frontend-a3b2c1d4`

---

### 2. aws_s3_bucket.frontend
```terraform
resource "aws_s3_bucket" "frontend" {
  bucket        = "${var.project_name}-frontend-${random_id.bucket_suffix.hex}"
  force_destroy = true
}
```
**目的**: フロントエンドファイルを格納するS3バケットを作成

**設定項目**:
- `bucket`: バケット名 = プロジェクト名 + "-frontend-" + ランダムサフィックス
- `force_destroy = true`: バケット内にオブジェクトがある状態でも削除を許可
  - ⚠️ 本番環境では`false`にすることを推奨（誤削除防止）
  - 開発環境では`true`が便利（`terraform destroy`でクリーンアップ可能）

---

### 3. aws_s3_bucket_website_configuration.hosting
```terraform
resource "aws_s3_bucket_website_configuration" "hosting" {
  bucket = aws_s3_bucket.frontend.id
  index_document {
    suffix = "index.html"
  }
}
```
**目的**: S3バケットを静的ウェブサイトとして設定

**設定項目**:
- `index_document.suffix = "index.html"`: デフォルトドキュメント（トップページ）を指定
- この設定により、`http://bucket-name.s3-website-region.amazonaws.com/`にアクセスするとindex.htmlが表示される
- オプション: `error_document`を追加して、404エラーページもカスタマイズ可能

---

### 4. aws_s3_bucket_public_access_block.public_access
```terraform
resource "aws_s3_bucket_public_access_block" "public_access" {
  bucket = aws_s3_bucket.frontend.id

  block_public_acls       = false
  block_public_policy     = false
  ignore_public_acls      = false
  restrict_public_buckets = false
}
```
**目的**: パブリックアクセスブロック設定を無効化（公開ウェブサイトに必要）

**設定項目** (すべて`false`):
- `block_public_acls`: パブリックACLの設定をブロックしない
- `block_public_policy`: パブリックバケットポリシーの設定をブロックしない
- `ignore_public_acls`: 既存のパブリックACLを無視しない
- `restrict_public_buckets`: パブリックポリシーを持つバケットへのアクセスを制限しない

**セキュリティ考慮事項**:
- 公開ウェブサイトホスティングには必要だが、セキュリティリスクあり
- 機密情報を含むファイルは絶対にアップロードしないこと
- CloudFrontなどのCDNを併用し、S3への直接アクセスを制限することを推奨

---

### 5. aws_s3_bucket_policy.allow_public_read
```terraform
resource "aws_s3_bucket_policy" "allow_public_read" {
  depends_on = [aws_s3_bucket_public_access_block.public_access]
  bucket     = aws_s3_bucket.frontend.id
  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Sid       = "PublicReadGetObject"
        Effect    = "Allow"
        Principal = "*"
        Action    = "s3:GetObject"
        Resource  = "${aws_s3_bucket.frontend.arn}/*"
      },
    ]
  })
}
```
**目的**: バケット内のすべてのオブジェクトに対する読み取り権限を全員に付与

**ポリシーの詳細**:
- `depends_on`: パブリックアクセスブロック解除後にポリシーを適用
- `Principal = "*"`: すべてのユーザー（インターネット上の誰でも）
- `Effect = "Allow"`: 許可
- `Action = "s3:GetObject"`: オブジェクトの取得のみ許可（一覧表示や削除は不可）
- `Resource = "${aws_s3_bucket.frontend.arn}/*"`: バケット内のすべてのオブジェクト

**セキュリティ注意点**:
- 書き込み権限は付与していない（安全）
- 読み取り専用なので、静的サイトホスティングには適切
- より安全な構成として、CloudFront OAC (Origin Access Control)の使用を検討

---

### 6. aws_s3_object.index_html
```terraform
resource "aws_s3_object" "index_html" {
  bucket       = aws_s3_bucket.frontend.id
  key          = "index.html"
  source       = "${path.module}/src/frontend/index.html"
  content_type = "text/html"
  etag         = filemd5("${path.module}/src/frontend/index.html")
}
```
**目的**: フロントエンドのHTMLファイルをS3にアップロード

**設定項目**:
- `bucket`: アップロード先バケット
- `key = "index.html"`: S3上でのファイル名（オブジェクトキー）
- `source`: ローカルのファイルパス
- `content_type = "text/html"`: MIMEタイプを明示的に指定（ブラウザで正しく表示されるため）
- `etag = filemd5(...)`: ファイルのMD5ハッシュを計算
  - ファイル内容が変更された場合のみアップロードを実行（差分更新）
  - `terraform apply`の効率化

**拡張方法**:
```terraform
# 追加のファイルをアップロードする場合
resource "aws_s3_object" "styles_css" {
  bucket       = aws_s3_bucket.frontend.id
  key          = "styles.css"
  source       = "${path.module}/src/frontend/styles.css"
  content_type = "text/css"
  etag         = filemd5("${path.module}/src/frontend/styles.css")
}
```

## アーキテクチャ全体での役割

このS3設定は、サーバーレスアプリケーションの**フロントエンド層**を構成します：

```
[ユーザー] 
   ↓ HTTP(S)
[S3 Static Website] ← このファイルで定義
   ↓ API Call
[API Gateway] (api_gateway.tfで定義)
   ↓
[Lambda] (lambda.tfで定義)
```

## ベストプラクティス

### 1. CloudFrontとの統合（推奨）
現在の構成では、S3ウェブサイトエンドポイントを直接公開していますが、以下の理由でCloudFrontの利用を推奨します：

- **HTTPS対応**: S3ウェブサイトエンドポイントはHTTPSをサポートしていない
- **パフォーマンス**: グローバルなエッジロケーションでのキャッシング
- **セキュリティ**: OACを使用してS3への直接アクセスを防止
- **コスト削減**: データ転送コストの最適化

### 2. バージョニングの有効化（推奨）
```terraform
resource "aws_s3_bucket_versioning" "frontend_versioning" {
  bucket = aws_s3_bucket.frontend.id
  versioning_configuration {
    status = "Enabled"
  }
}
```
誤った更新からの復旧が可能になります。

### 3. ライフサイクルポリシー
古いバージョンの自動削除でコスト管理が可能です。

### 4. ログの有効化
```terraform
resource "aws_s3_bucket_logging" "frontend_logging" {
  bucket        = aws_s3_bucket.frontend.id
  target_bucket = aws_s3_bucket.logs.id
  target_prefix = "frontend-access-logs/"
}
```
アクセスログの記録でトラブルシューティングが容易になります。

## デプロイ後の確認

1. **ウェブサイトURL取得**:
   ```bash
   terraform output website_url
   ```

2. **S3コンソールで確認**:
   - プロパティ → 静的ウェブサイトホスティング → エンドポイント

3. **ブラウザでアクセステスト**:
   - index.htmlが正しく表示されることを確認
   - ブラウザの開発者ツールでAPI呼び出しが成功することを確認

## トラブルシューティング

### 問題: 403 Forbiddenエラー
- バケットポリシーが正しく適用されているか確認
- パブリックアクセスブロック設定を確認
- オブジェクトのキー名が正しいか確認

### 問題: ファイルが更新されない
- `etag`を確認（ファイルハッシュが変更されているか）
- ブラウザキャッシュをクリア
- S3コンソールでオブジェクトの更新日時を確認

### 問題: CORSエラー
- API Gatewayのレスポンスヘッダーに`Access-Control-Allow-Origin`が含まれているか確認
- OPTIONSメソッド（プリフライトリクエスト）が正しく設定されているか確認
